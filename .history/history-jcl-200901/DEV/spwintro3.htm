３．フレームワーク構成

　Sparrow FrameworkはまずClient Application FrameworkとService Application Frameworkから構成されます。

　Client Application Frameworkは以下の部品から構成されています。

3.1 Javascript Object Framework

　　Javascript Object FrameworkはObjectをClass、Namespace、Instance、Externalに分類して、Javascriptのプロトタイプベースオブジェクト指向モデルを基づいてクラスベース指向のオブジェクト指向モデルを実現しています。クラスはプログラムの単位になり、クラスの継承などを利用してJavascriptでも複雑のフレームワーク及ぶアプリケーションを容易に作成することになります。

3.2 Component Framework

Component Frameworkはコンポーネントの基本機能をサポートします。

n         オーナーシップ機能

他のコンポーネントを管理する機能。コンポーネント A がコンポーネント B のオーナーである場合，A は A が破棄されるときに B も破棄します。

n         持続化

コンポーネントをXMLにシリアライズしたり、またXMLからコンポーネントを生成することができます。

n         イベント処理

コンポーネントではイベントを起こすことができます。コンポーネントのイベントのリスナーを登録しておけば、コンポーネントのイベントが発生した時にそれをキャッチし、処理できます。

3.3 Widget Framework

　　Widget Frameworkはグラフィック関連のコンポーネント(コントロール)群を提供しています。コントロールには親子関係を持っていて、コントロールは親コントロール（コンテナーという）のクライアント領域に配置されます。親子階層ツリーでRootとなっているコンテナーはDesktopといいます。

　  Widget Frameworkでも軽いMVC構造をもっています。UIエレメント 、UIコントローラ はそれぞれV、Cの役割を担当して、コントロール自体はMとなります。

UIエレメントはHTMLエレメントにマッピングされ、画面のレンダリングをします。

UIコントローラはHTMLのイベントをインテリジェンスに管理し、適当なControlに振り分けます。

コントロールはプロパティ値の変更をUIエレメントに反映したり、UIコントローラからメッセージを処理したりします。

3.4 Dataset Framework

Datasetは、アプリケーションで使用するためのビジネスデータオブジェクトを格納できるデータテーブルを含むオブジェクトです。

アプリケーションでデータを操作する必要がある場合、サーバ上のData Serviceでデータベースからデータを読み込んで、ビジネスロジックによって処理してServer Datasetに格納します。

Server Datasetのデータは HTTPでクライアントに送信されClient Datasetに格納されます。

クライアント側でClient Datasetのデータをコントロールに表示して、またユーザがUIコントロールを操作することによるデータへの変更はClient Datasetに保持されるので、その変更がHTTPでServer Datasetに反映され、Data Serviceに伝わります。

Datasetの構造はリレーショナル データベースに似ていて、テーブル、行、列、制約、およびリレーションシップの階層オブジェクトモデルを公開します。　

3.5 Service Framework

　　クライアント側のサービスはローカルで実行するLocal ServiceとリモートのData ServieへのアダプタのRemote Service Adapterから構成されます。

3.6 Application Framework

　　Widget Framework、Dataset Framework、Service Frameworkはそれぞれ独立性をもっています。これら各部分を統合したApplication FrameworkはApplicationにおける各Form、Dataset、Serviceなどのリソースをインテリジェンスに管理する機能を提供します。
